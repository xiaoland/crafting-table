## Purpose

This packet captures the next independent capability under consideration: local LLM inference hosted by CraftingTable on iPad.

The feature should let CraftingTable act as a local model host for nearby tools. The first useful version should be practical and measurable: manage one downloaded open-source model, run it through a native runtime, and expose a small HTTP surface that OpenAI Responses API clients can adapt to with minimal glue.

## Product Direction

The first product promise is a foreground local model server for trusted LAN use.

Minimum user-visible behavior:

- The app shows the active HTTP server address, including LAN IP and port.
- The app shows whether the server is stopped, starting, listening, generating, or failed.
- The app shows the active model and model readiness state.
- The user can download, verify, activate, switch, and remove models through a Model Manager.
- LAN clients can call the service with a bearer token generated by the app.

Explicit first-slice limits:

- The service is foreground-only.
- The app displays the listening address directly.
- Automatic server discovery is deferred.
- Streaming is deferred.
- Multimodal input, tools, background responses, and durable conversation state are deferred.

## Runtime Direction

Use `llama.cpp + GGUF + Metal` as the first runtime.

Rationale:

- GGUF is a single-file model format used by GGML and llama.cpp.
- Hugging Face has native GGUF browsing and metadata support.
- Public GGUF model files can carry size and sha256 metadata suitable for a simple Model Manager.
- llama.cpp has a direct path to Metal acceleration on Apple platforms.

Keep the runtime boundary explicit:

- `LocalLLMRuntime` owns load, unload, generate, cancel, and runtime status.
- `LlamaCppRuntime` is the first implementation.
- MLC LLM remains a later adapter after the first GGUF path proves the product need.

## HTTP API Direction

Expose the smallest OpenAI Responses-compatible HTTP surface:

- `GET /health`
- `GET /v1/models`
- `POST /v1/responses`

First `POST /v1/responses` support:

- Accept `model`, `input`, `instructions`, `max_output_tokens`, `temperature`, and `top_p` when practical.
- Return a synchronous JSON response shaped like an OpenAI Response object.
- Project unsupported fields into ignored metadata or explicit validation errors.
- Map the active local model to an OpenAI-style model id.

The server should bind to a LAN-reachable interface when enabled by the user. The UI should show the concrete URL, for example `http://192.168.1.42:8787/v1/responses`.

## Security Direction

Use HTTP Bearer Authorization for the first slice.

First security shape:

- Generate a high-entropy opaque bearer token on the iPad.
- Store the token in the platform credential store when implementation begins.
- Require `Authorization: Bearer <token>` on all model and response endpoints.
- Allow token rotation from the app UI.
- Keep the service disabled until the user explicitly starts it.
- Add `NSLocalNetworkUsageDescription` when LAN server implementation begins.

JWT remains a later option when the server needs signed claims, expiry, or multiple clients with distinct scopes.

## Model Manager Direction

The Model Manager owns model discovery, download, verification, cache, activation, switching, and deletion.

First model record fields:

- `id`
- `displayName`
- `source`
- `repositoryID`
- `revision`
- `filename`
- `downloadURL`
- `license`
- `fileSize`
- `sha256`
- `localPath`
- `downloadState`
- `verificationState`
- `activationState`
- `runtimeCompatibility`
- `createdAt`
- `updatedAt`

First behavior:

- Download with progress.
- Resume partial downloads when the platform download mechanism supports it.
- Verify sha256 before activation.
- Keep verified files in app support storage.
- Treat a verified file plus source metadata as the cache key.
- Allow one active model at a time.

## Model Source Investigation

Hugging Face should be the first source provider.

Useful observed Hub paths:

- GGUF browsing: `https://huggingface.co/models?library=gguf`
- Model list API: `https://huggingface.co/api/models?filter=gguf`
- Repo info API: `https://huggingface.co/api/models/{repo_id}?blobs=true`
- File download URL: `https://huggingface.co/{repo_id}/resolve/{revision}/{filename}`

Useful observed Hugging Face metadata:

- model id
- repo commit sha
- tags, including license tags
- card data, including license and base model when present
- sibling files
- GGUF sibling size
- LFS sha256 for large files when `blobs=true`
- HEAD response headers such as `X-Repo-Commit`, `X-Linked-Size`, and `X-Linked-ETag`

GitHub Releases can become a second provider for curated manifests or user-supplied release assets. Release assets expose download URLs, size, and sha256 digest when available.

Custom URL import should remain in scope as a pragmatic escape path after the Hugging Face provider shape is proven.

## Scope

In scope for the first implementation slice:

- define the local runtime boundary
- integrate llama.cpp enough to load and generate with one GGUF model
- create a Model Manager data model and storage path
- add Hugging Face GGUF source lookup for a curated or searched model
- download and verify one model
- start and stop a foreground LAN HTTP server
- require bearer authorization
- implement synchronous `POST /v1/responses`
- implement `GET /v1/models`
- show listening URL, token status, server state, and active model state

Deferred scope:

- streaming responses
- tool calls
- multimodal inputs
- background service claims
- automatic LAN discovery
- multi-client identity and per-client authorization scopes
- model quantization inside the app
- broad model search ranking
- runtime benchmarking UI
- MLC LLM runtime adapter
- Core ML or Apple Foundation Models integration

## Initial Execution Slices

1. Product and contract solidification
   - Confirm the first OpenAI Responses subset.
   - Confirm security UX for bearer token display and rotation.
   - Confirm first curated GGUF model candidates.

2. Model source spike
   - Query Hugging Face for GGUF repos.
   - Fetch repo info with blob metadata.
   - Build a local model manifest candidate from one public ungated GGUF file.

3. Model Manager spike
   - Download one GGUF file.
   - Verify sha256.
   - Persist local model metadata.
   - Activate and switch the current model record.

4. Runtime spike
   - Build or link llama.cpp for iOS with Metal.
   - Load the verified GGUF file.
   - Generate one short response from a local prompt.

5. HTTP server spike
   - Start a foreground LAN listener.
   - Enforce bearer authorization.
   - Serve `GET /health`, `GET /v1/models`, and `POST /v1/responses`.
   - Call the endpoint from another LAN device.

6. UI integration spike
   - Show server address, bearer token state, model state, download state, and active model.
   - Keep the UI operational while generation is running.

## Verification Plan

- Confirm Hugging Face source lookup returns a public ungated GGUF candidate with file size and sha256.
- Download the selected file and compare computed sha256 with source metadata.
- Confirm corrupted or incomplete downloads cannot be activated.
- Load the verified GGUF model through llama.cpp on an iPad-capable build target.
- Generate a short response from a local prompt.
- Start the foreground HTTP server and confirm it binds to a LAN-reachable address.
- Confirm unauthenticated requests are rejected.
- Confirm authenticated `GET /v1/models` returns the active model.
- Confirm authenticated `POST /v1/responses` returns a synchronous text response.
- Confirm another LAN device can call the iPad URL while the app remains foregrounded.
- Stop the server and confirm requests fail cleanly.
