# Early product truths

## Purpose

This file preserves the small set of product truths that appear stable enough to constrain future exploration.

It is intentionally narrow.

## Product center

- `Xiaoland Workbench` is a personal, iPad-first central control surface for life and work, with AI in the loop.
- The product optimizes for the repo owner's personal utility rather than generic market breadth.
- The product is not developer-only and should not collapse into an agent-centric tool.
- The first-class execution object is the `work session`, but the product itself is broader than any single session workflow.
- `Goal Forest` is a longer-lived orientation surface around sessions and captures rather than a replacement for `work session`.
- Goals, captured fragments, tasks, tools, and reviews should support orientation and execution across the broader control surface.

## Canonical language

- `Goal Forest` is currently a canonical user-facing term for the shared workspace.
- Because this term is metaphorical, the product should pair it with literal operational language when clarity matters, especially around sessions, actions, and review states.
- The metaphor should mainly name the surface, not replace literal action labels.

## Durable product pressures

- The product should help the user regain orientation, choose focus, and act from one shared surface.
- Capture should stay cheap even when the user does not yet know where something belongs.
- The system should reduce classification tax by separating raw intake from later structure.
- Humans and agents should read from and write to the same shared workspace.
- The product should surface the subset of state that matters now instead of forcing the user to traverse the full graph every time.
- Early versions should stay legible and minimal while the core loop is still being proven.

## Core tool surfaces

- Terminal-based remote machine control is a core tool surface for the target user, not merely an adjacent convenience.
- Remote terminal access and file transfer belong to the same primary workflow.
- Remote control should strengthen the broader control surface and active work sessions rather than live as an unrelated utility surface.

## Scope discipline

- A tool surface belongs in the product only if it materially improves orientation, execution, review, or continuity on the shared control surface.
- Important tool surfaces may exist without becoming the product's sole core loop.

## Scope boundaries for now

- Do not treat a tree, graph, or any other storage shape as durable product truth in the PRD layer yet.
- Do not assume broad integrations or autonomous background restructuring as first-version requirements.
- The first remote shape is terminal-first and file-transfer-capable rather than GUI remote desktop.
- Do not assume agent-operable `Goal Forest` or remote-control workflows, integrated agent-output review, or Pencil-specific workflows as first-version requirements.
- Any additional action surface must earn its place by strengthening the main work loop rather than creating a second product.
