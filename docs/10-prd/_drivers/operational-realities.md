# Operational realities

## Realities

- existing system limitation: the repo is still early, so docs must guide growth without pretending the full architecture is known.
  runtime reality: product truth is stabilizing faster than code structure.
  migration dependency: durable product and cross-unit technical memory need to exist before implementation fans out.

- existing system limitation: `Goal Forest` is metaphorical language.
  runtime reality: the metaphor adds identity but can reduce clarity if not paired with literal operational terms.
  migration dependency: vocabulary must stay stable as screens and code names begin to appear.

- existing system limitation: remote machine control can easily drift into a standalone utility.
  runtime reality: the target user already has real remote workflows that matter.
  migration dependency: Remote Control must stay linked to work sessions and broader orientation during implementation.

- existing system limitation: codex-app server details can churn across hosts and Codex releases.
  runtime reality: a small host-side Codex Remote Server can absorb protocol and platform differences while CraftingTable keeps a stable contract.
  migration dependency: Codex Remote should preserve a separate boundary from Remote Control, Goal Forest, and Work Session until integration pressure is proven.
